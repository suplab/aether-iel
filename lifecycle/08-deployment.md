# Phase 8 — Deployment

> **AIEL Phase Group: Intelligence Operations**
> Phase 8 takes the evaluated system into production. It is not a handover — the intelligence team remains accountable through this phase. Deployment is complete only when observability is confirmed and the system is demonstrably operating within its defined parameters.

---

## Purpose

Deploying an intelligence system is qualitatively different from deploying conventional software. A conventional deployment is successful when the application is running and returning correct HTTP responses. An intelligence deployment is successful when: the system is running, observable, operating within its confidence and latency parameters, and the human review queue is receiving the expected volume of escalations.

Phase 8 ends when these conditions are confirmed — not when the pods are running.

---

## Pre-Deployment Checklist

All of the following must be in place before any production deployment:

**Infrastructure**
- [ ] PostgreSQL instance with pgvector extension installed and ivfflat index created
- [ ] Ollama (or equivalent embedding service) deployed and reachable from the application
- [ ] LLM API credentials configured as secrets (not environment variables in plaintext)
- [ ] Secret rotation policy configured for all API keys
- [ ] Network policy configured — the application must not be reachable from the internet without authentication

**Observability**
- [ ] Prometheus metrics endpoint (`/actuator/prometheus`) is running and scraped
- [ ] Grafana dashboards are deployed: agent invocations by type, confidence score distribution, human review queue depth, p99 latency by endpoint, memory operations per minute, decay job execution status
- [ ] Alerting rules are configured: p99 > 1500ms (warn), p99 > 2000ms (critical), human review queue depth > N (warn), confidence gate failures (any auto-enforced action below 0.8 — this should never happen but must alert if it does)
- [ ] Centralized logging is configured with structured JSON format
- [ ] Distributed tracing is configured (OpenTelemetry)

**Security**
- [ ] Authentication is enforced on all API endpoints
- [ ] Authorization: memory endpoints are scoped to the authenticated user's `user_id` — no cross-user data access
- [ ] TLS is enforced on all external endpoints
- [ ] OWASP scan was clean in Phase 7 — no new dependencies added since then

**Data**
- [ ] Flyway migrations have run on the production schema
- [ ] No test data or evaluation data in the production database
- [ ] GDPR consent model is in place — the system does not process memory unless consent is recorded

## Deployment Strategy

Intelligence systems require a more conservative deployment strategy than conventional software because failures are less deterministic and harder to detect.

**Canary deployment** is the required strategy for any system at IMM Level 3 or above. Route a small fraction of traffic (10%) to the new system version. Monitor confidence score distribution, human review queue depth, and latency for 24 hours before expanding. If the confidence distribution shifts significantly (the system is suddenly less certain) or the human review queue depth increases unexpectedly, roll back.

**Feature flags** for new agent capabilities. A new agent or a new authority level should be introduced behind a feature flag and enabled for a subset of users before full rollout. This allows the authority expansion to be measured before it affects the full user population.

**Database migration ordering**: schema migrations run before the application deployment. The old application version must be compatible with the new schema. If a migration removes a column or table that the old version reads, the migration and the deployment must be sequenced carefully (expand-contract pattern).

---

## Rollback Plan

Every deployment must have a documented rollback plan executed by a named owner. The rollback plan for an intelligence system includes:

*Application rollback*: revert to the previous container image. This is the same as conventional software.

*Schema rollback*: if the deployment includes schema migrations, the rollback must include reverse migrations. Flyway does not support automatic rollback — reverse migrations must be written and tested before the deployment.

*Memory state*: if the deployment changes how memories are stored (new fields, new types), rolling back the application does not roll back memory state. Document how memories written by the new version will be handled by the old version — either they are compatible (preferred), or the incompatible memories must be cleaned up as part of rollback.

*Human review queue*: if the system is rolled back, decisions in the human review queue that were generated by the new version may have a different format. Document how these are handled.

## Post-Deployment Validation

Immediately after deployment, before declaring the deployment successful:

1. Run the smoke test suite against the production system — a set of end-to-end requests that exercise all major agent types and verify correct responses.
2. Verify the confidence gate is operating: submit a request designed to produce low confidence and confirm it appears in the human review queue.
3. Verify memory operations: create a memory, retrieve it by similarity, confirm it is returned.
4. Verify observability: confirm that agent invocation metrics are appearing in Prometheus, that the Grafana dashboards are populated, and that the alerting rules are active.
5. Check the human review queue depth: confirm it is within expected parameters. A queue depth of zero means the confidence gate may not be routing correctly.

---

## Phase Gate

Deployment is complete when:

- [ ] Application is running and passing health checks (`/actuator/health`)
- [ ] Smoke test suite passes against production
- [ ] Confidence gate validation confirms low-confidence routing is working
- [ ] Memory operations are verified end-to-end
- [ ] All Grafana dashboards are populated with live data
- [ ] Alerting rules are confirmed active
- [ ] GDPR consent model is verified in production
- [ ] Human review queue depth is within expected range
- [ ] Deployment sign-off is recorded
